# 工程知识卡（Stage-1）

## 1. 项目定义

项目名称：冰箱食材识别与管理系统。

当前仓库处于 **Stage-1 本地闭环验证阶段**，目标不是一次性完成全量智能冰箱系统，而是先证明“视觉事件检测 → 事件结构化输出 → Python 接入 → SQLite 库存更新 → 本地页面展示/人工修正”这条最小闭环可以稳定跑通。

## 2. 当前阶段目标

Stage-1 的明确目标如下：

1. 从本地视频读取冰箱交互过程。
2. 检测真实冰箱交互事件。
3. 导出结构化 `event.json`。
4. 由 Python 后端接入事件。
5. 更新 SQLite 库存记录。
6. 在本地网页展示库存、近期事件、待确认项与人工修正入口。

这意味着当前工程优先保证：

- 系统闭环成立；
- 真实事件识别链路可复现；
- 库存状态能够更新；
- 存在模糊事件时能够进入人工确认闭环；
- 代码、文档、测试可以支撑比赛演示。

## 3. 当前阶段不追求的内容

以下内容属于后续扩展，不是 Stage-1 的主交付目标：

- 复杂细粒度食材识别大模型
- 多模态云端重识别闭环
- 完整嵌入式板端部署适配
- 实时摄像头流替代全部本地文件输入
- 多对象高精度实例级跟踪

当前仓库明确将真实轻量分类器、实时视频流、板端适配都保留为后续 TODO。

## 4. 设计原则

### 4.1 分层识别、逐级确认

当前视觉链路遵循：

1. 先做事件识别；
2. 再做物体存在性判断；
3. 最后再接细粒度分类。

这与比赛中“先保证真实事件检测与库存闭环，再扩展识别能力”的路线一致。

### 4.2 先保基础得分，再做扩展

当前工程优先覆盖以下评审得分点：

- **系统完整性**：前后端闭环跑通；
- **真实事件识别**：区分放入、取出、无变化、疑似部分取出；
- **库存管理**：事件驱动库存更新；
- **部分取出处理**：通过候选事件 + 待确认机制降低库存漂移；
- **可演示性**：本地页面可查看状态并进行人工修正。

## 6. 系统主流程

### 6.1 视觉侧主流程

本地视频 / 帧目录输入 → 运动摘要计算 → 交互窗口定位 → `before/after` 关键帧选择 → ROI 差异分析 → 输出事件类型 → 生成 `event.json` 与调试图。

当前输出事件类型仅包含四类：

- `no_change`
- `put_in`
- `take_out`
- `partial_take_out_candidate`

### 6.2 后端主流程

扫描 `data/sessions/**/event.json` → 导入尚未处理事件 → 写入事件历史 → 更新 SQLite 库存 → 返回 Web/API 展示结果 → 支持待确认与人工修正。

### 6.3 演示闭环

1. 运行 C++ 视觉程序；
2. 生成 `data/sessions/<session_id>/event.json`；
3. 刷新或访问 Python 本地页面；
4. 后端自动导入新事件；
5. 页面展示库存、事件、待确认项；
6. 用户对模糊事件进行确认或修正。

## 7. 当前关键接口协议

核心共享协议文件：`shared/event_schema.json`

### 7.1 顶层关键字段

- `session_id`
- `timestamp`
- `event_type`
- `roi_id`
- `confidence`
- `before_frame`
- `after_frame`
- `change_regions`
- `objects`
- `need_user_confirm`

### 7.2 objects 关键语义

每个对象条目当前包含：

- `category`：粗分类类别
- `name`：对象名称
- `count_delta`：数量变化
- `remain_level`：剩余量估计（0~1）

### 7.3 协议作用

该协议是当前工程最关键的跨模块边界，负责将 C++ 视觉结果稳定传递到 Python 库存系统。后续所有扩展都应尽量兼容此协议，避免前后端接口频繁漂移。

## 8. 当前视觉链路的启发式核心

当前 C++ 视觉链路仍以事件检测为中心，而不是完整识别模型。核心指标包括：

- `changed_ratio`：ROI 变化占比
- `mean_delta`：变化区域平均有符号亮度差
- `peak_transition`：最强交互转折点
- `stable_frame_score`：关键帧稳定性评分

当前逻辑重点是：

- 变化很低时判 `no_change`
- 有效变化且符号证据偏负时倾向 `put_in`
- 有效变化且符号证据偏正时倾向 `take_out`
- 变化明显但证据不足时判 `partial_take_out_candidate`

这是符合比赛“先做真实事件判断，再处理复杂识别”的工程路线的。

## 11. 评审得分点映射

### 11.1 系统完整性

由以下链路支撑：

- C++ 事件检测输出
- `event.json` 协议
- Python 事件接入
- SQLite 库存更新
- 本地页面展示与人工修正

### 11.2 真实事件识别

由视觉主链支撑：

- ROI 运动分析
- 关键帧选择
- `put_in / take_out / no_change / partial_take_out_candidate` 事件判定

### 11.3 物体分类能力

当前仅保留粗粒度对象字段与后续扩展接口，真实分类器暂未接入。

比赛演示时应避免夸大为“已完成高精度细粒度识别系统”，应如实表述为：当前阶段已完成事件识别闭环，并为后续轻量分类器接入预留接口。

### 11.4 库存管理策略

由后端库存服务支撑：

- 事件驱动更新
- 当前库存展示
- 近期事件历史
- 待确认处理
- 手动修正

### 11.5 部分取出处理

当前通过 `partial_take_out_candidate` + `need_user_confirm` 路线实现，是 Stage-1 条件下较稳妥的工程折中方案。

## 12. 已识别风险

当前已知主要风险包括：

- 本地工具链对视频解码能力依赖较强；
- 低照度场景下阈值不稳定；
- 手部遮挡、整理食材等动作可能触发伪变化；
- 没有真实分类器时，部分取出与普通取出存在歧义；
- 若人工修正链路缺失，库存状态会逐步漂移。

## 13. 当前建议的开发优先级

### P0：必须先稳定

- C++ 事件检测主链可重复跑通
- `event.json` 协议冻结

- 页面能够展示库存、事件、待确认项
- 人工确认与手动修正闭环成立

### P1：优先增强

- 低照度、遮挡、手部干扰场景测试
- 参数配置化与调试结果可视化
- 部分取出判定规则优化
- 后端幂等、异常保护、测试补齐

### P2：后续扩展

- 接入轻量粗分类模型
- 替换实时摄像头流
- 迁移到嵌入式板端
- 增加多对象场景与包装信息利用

## 14. 对新对话 / 新成员 / 新 AI 的使用说明

阅读本知识卡时，应默认接受以下事实：

1. 当前项目不等于完整商用智能冰箱，而是比赛阶段性工程原型；
2. 当前优先目标是 Stage-1 本地闭环，不是盲目追求大模型识别；
3. 任何新增方案都不得破坏现有 `event.json` 主接口；
4. 任何扩展都应优先服务于比赛评审得分点；
5. 对外表述时要区分“已完成”“预留接口”“后续扩展”，避免叙述失真。

## 15. 后续维护要求

当以下内容发生变化时，必须同步更新本知识卡：

- 目录结构有明显调整；
- 事件协议字段变更；
- Stage-1 目标升级为 Stage-2；
- 接入真实分类器；
- 后端接口或库存规则变更；
- 嵌入式部署路径确定。
