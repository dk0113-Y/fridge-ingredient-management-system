# 工程知识卡（New Baseline）

## 1. 项目定义

项目名称：冰箱食材识别与管理系统。

当前仓库在保留原有 **Stage-1 本地闭环验证能力** 的基础上，工程脉络升级为 **“事件触发 + 目标检测与变化匹配 + 类别识别 + 库存决策”** 的分层式架构。

新版基线的核心目标不是推翻现有工程，而是在不破坏既有闭环的前提下，将视觉前端从“单一事件启发式主导”升级为“分层感知、逐级确认、最终由库存决策层统一输出”的结构化流水线。

## 2. 当前阶段目标

新版工程基线的当前目标如下：

1. 继续保留现有真实交互事件触发能力；
2. 在 `before/after` 稳定帧上引入轻量目标检测；
3. 通过前后帧目标匹配判断新增、消失与重排；
4. 仅对真正发生增减的目标执行类别识别；
5. 由库存决策层统一输出 `put_in / take_out / no_change / partial_take_out_candidate`；
6. 继续通过 Python + SQLite + Web 完成库存闭环、待确认与人工修正。

这意味着当前工程优先保证：

- 原有最小闭环不被破坏；
- 真实事件检测仍然稳定；
- 目标“有无变化”和“数量变化”判断更可靠；
- 类别识别与库存决策职责解耦；
- 离线情况下仍可依靠端侧粗分类维持基础闭环；
- 模糊事件继续进入待确认闭环而不是强行误判。

## 3. 当前阶段不追求的内容

以下内容仍属于后续扩展，不作为当前主交付目标：

- 完整细粒度食材识别全覆盖
- 以云端大模型为主依赖的识别链路
- 全量实时多摄像头视频流系统
- 板端全链路深度适配与性能压测闭环
- 多对象复杂实例跟踪与长期身份保持

当前工程明确将“云端大模型/VLM”定义为**可选增强模块**，而不是基础闭环的必需条件。

## 4. 设计原则

### 4.1 分层识别、逐级确认

新版视觉与决策链路遵循：

1. 先做真实事件触发；
2. 再做前后帧目标检测；
3. 再做新增/消失/重排匹配；
4. 对确有增减的目标做类别识别；
5. 最终由库存决策层统一给出库存动作与确认状态。

该结构与比赛强调的“先保证真实事件检测与系统闭环，再扩展物体识别与类别能力”的路线一致。

### 4.2 先保基础得分，再做扩展

当前工程优先覆盖以下评审得分点：

- **系统完整性**：事件触发、识别、库存更新、展示、修正闭环完整；
- **真实事件识别**：区分真实交互、无变化、放入、取出、部分取出候选；
- **物体存在与数量变化判断**：通过前后帧检测与匹配提高可靠性；
- **库存管理**：由统一决策层驱动库存更新，降低漂移；
- **部分取出处理**：保留候选事件与人工确认机制；
- **可演示性**：本地页面可展示状态并对模糊结果进行修正。

### 4.3 类别识别与库存决策解耦

类别识别模块只负责“这是什么”，不直接负责“库存如何更新”。

库存决策必须结合：

- `before/after` 检测结果；
- 框匹配关系；
- 类别结果；
- 历史库存；
- 规则置信度；
- 是否需要人工确认。

## 5. 新版架构层次

### 5.1 第一层：事件触发层

职责：判断是否发生真实冰箱交互，并定位事件窗口。

当前实现应继续沿用既有逻辑：

- 运动摘要计算
- ROI 限定
- 峰值转折点定位
- `before/after` 关键帧选择
- 稳定帧评分

该层输出的核心不是类别，而是：

- 是否触发事件
- 事件时间窗口
- `before_frame`
- `after_frame`
- 候选变化区域

### 5.2 第二层：目标检测与变化匹配层

职责：在 `before/after` 稳定帧上分别检测物体，并通过前后匹配判断哪些目标是新增、消失或仅重排。

该层推荐流程如下：

1. 在 `before_frame` 上运行轻量检测器，得到 `before_objects`；
2. 在 `after_frame` 上运行轻量检测器，得到 `after_objects`；
3. 基于 IoU、类别粗标签、尺寸相近性、位置容忍度执行匹配；
4. 输出三类结果：
   - `matched_objects`：前后均存在，仅位置/姿态变化；
   - `added_objects`：仅在 `after` 出现；
   - `removed_objects`：仅在 `before` 出现。

该层只负责“变化结构化”，不直接做最终入库/出库决策。

### 5.3 第三层：类别识别层

职责：仅对真正有必要识别的目标裁剪图进行类别识别。

当前推荐策略：

- 对 `added_objects` 使用 `after_frame` crop 识别；
- 对 `removed_objects` 使用 `before_frame` crop 识别；
- 对 `matched_objects` 默认不做细分类，除非存在低置信度或规则冲突；
- 端侧必须具备粗分类能力；
- 云端大模型/VLM 只作为细分类增强，不可成为基础链路唯一依赖。

当云端细分类不可用时，系统应继续使用端侧粗分类结果完成基础闭环，并将该条目标记为待确认。

### 5.4 第四层：库存决策层

职责：综合第二层与第三层结果、历史库存和规则置信度，统一输出库存动作。

该层输出应包括但不限于：

- `put_in`
- `take_out`
- `no_change`
- `partial_take_out_candidate`
- `need_user_confirm`

该层是比赛中“库存管理、部分取出处理、系统闭环稳定性”的核心得分支撑层。

## 6. 系统主流程

### 6.1 新版视觉与决策主流程

本地视频 / 帧目录输入 → 事件触发 → `before/after` 稳定帧提取 → 轻量目标检测 → 前后匹配 → 识别新增/消失/重排 → 对新增/消失目标执行类别识别 → 库存决策层输出事件 → 生成 `event.json` 与调试图。

### 6.2 后端主流程

扫描 `data/sessions/**/event.json` → 导入尚未处理事件 → 写入事件历史 → 更新 SQLite 库存 → 返回 Web/API 展示结果 → 支持待确认与人工修正。

### 6.3 演示闭环

1. 运行 C++ 视觉主程序；
2. 生成 `data/sessions/<session_id>/event.json`；
3. 刷新或访问 Python 本地页面；
4. 后端自动导入新事件；
5. 页面展示库存、事件、待确认项；
6. 用户对模糊事件或待确认条目进行确认与修正。

## 7. 当前关键接口协议

核心共享协议文件仍为：`shared/event_schema.json`

### 7.1 协议冻结原则

当前最重要原则仍然是：**新版架构升级不得轻易破坏现有 `event.json` 主接口。**

顶层关键字段仍建议保留：

- `session_id`
- `timestamp`
- `event_type`
- `roi_id`
- `confidence`
- `before_frame`
- `after_frame`
- `change_regions`
- `objects`
- `need_user_confirm`

### 7.2 推荐新增的可选扩展字段

为适配新版分层链路，可在不破坏现有消费端的前提下新增可选字段：

- `before_objects`
- `after_objects`
- `matched_objects`
- `added_objects`
- `removed_objects`
- `classification_source`（如 `local_classifier` / `vlm` / `fallback`）
- `decision_trace`（用于调试库存决策来源）
- `inventory_action_confidence`

这些字段应视为可选增强字段，旧后端在忽略这些字段时仍应能正常工作。

### 7.3 objects 关键语义

每个对象条目仍建议包含：

- `category`：粗分类类别
- `name`：对象名称
- `count_delta`：数量变化
- `remain_level`：剩余量估计（0~1）

必要时可扩展：

- `source_frame`：来自 `before` 还是 `after`
- `track_status`：`matched / added / removed`
- `is_provisional`：是否为待确认条目

## 8. 新版视觉链路的核心逻辑

新版 C++ 视觉主链不再由单一启发式直接输出全部库存动作，而是采用“启发式事件触发 + 检测匹配 + 决策融合”的组合方式。

### 8.1 事件触发启发式仍然保留

以下核心指标仍然有价值：

- `changed_ratio`
- `mean_delta`
- `peak_transition`
- `stable_frame_score`

这些指标仍然负责：

- 过滤无效事件；
- 定位 `before/after` 稳定帧；
- 为 `partial_take_out_candidate` 提供辅助证据；
- 降低不必要的检测与分类调用。

### 8.2 检测匹配是“数量变化判断”的核心

新版工程明确规定：

- 目标检测负责输出“看到了什么”；
- 目标匹配负责输出“哪些新增/消失/仅移动”；
- 库存决策层负责输出“库存如何更新”。

因此，系统不应将“库存增减判断”直接内嵌为检测器本身的职责。

### 8.3 部分取出仍需保留启发式与待确认机制

部分取出并不一定表现为目标完全消失，也可能表现为：

- 物体仍然存在但外观变化明显；
- 包装体仍在但内容量变化；
- 同一目标框存在但面积、纹理、液位或表观状态发生变化。

因此，`partial_take_out_candidate` 仍应由启发式证据、检测匹配结果和置信度共同决定，而不是简单等同于 `removed_objects`。

## 9. 当前后端职责

Python 后端继续负责：

- 事件 JSON 接入
- SQLite 库存状态维护
- 事件历史保存
- 待确认流程处理
- 手动库存修正
- 本地 Web/API 页面展示

当前最小接口继续包括：

- `GET /health`
- `GET /events`
- `GET /inventory`
- `POST /confirm`
- `GET /`

后端应逐步补充以下能力：

- 幂等导入保护
- provisional 条目展示与确认
- 决策追踪信息展示
- 冲突事件的人工修正入口

## 11. 评审得分点映射

### 11.1 系统完整性

由以下链路支撑：

- 事件触发
- 检测与变化匹配
- 类别识别
- 库存决策输出
- `event.json` 协议
- Python 事件接入
- SQLite 库存更新
- 本地页面展示与人工修正

### 11.2 真实事件识别

由第一层支撑：

- ROI 运动分析
- 关键帧选择
- 稳定帧提取
- 无效事件过滤

### 11.3 物体分类能力

由第二层与第三层共同支撑：

- 目标存在确认
- 新增/消失目标识别
- 端侧粗分类
- 可选云端细分类增强

对外表述时应如实说明：

- 端侧基础能力负责粗分类与闭环；
- 云端大模型/VLM 仅为增强项，不属于基础必需能力。

### 11.4 库存管理策略

由第四层与后端库存服务共同支撑：

- 检测匹配后的库存动作判定
- 当前库存展示
- 近期事件历史
- provisional / 待确认处理
- 手动修正

### 11.5 部分取出处理

由以下机制共同支撑：

- 启发式视觉证据
- 新增/消失/未消失但明显变化的目标分析
- `partial_take_out_candidate`
- `need_user_confirm`
- 页面确认与修正闭环

## 12. 已识别风险

当前已知主要风险包括：

- 低照度、反光和包装材质导致检测不稳定；
- 手部遮挡、整理动作与真实取放动作边界模糊；
- 多个相似物体互换位置时，匹配结果可能出错；
- 云端识别不可用时，若无本地粗分类兜底会导致闭环中断；
- 若 provisional 条目和人工修正链路缺失，库存状态会逐步漂移；
- 部分取出场景无法仅依赖目标消失判断。

## 13. 当前建议的开发优先级

- 保持现有事件触发主链可重复跑通

- 页面能够展示库存、事件、待确认项

### P1：优先增强

- 轻量检测器接入 `before/after` 双帧流程
- 前后目标匹配规则实现

- 低照度、遮挡、手部干扰场景测试
- 参数配置化与调试结果可视化
- 后端幂等、异常保护、测试补齐

### P2：后续扩展

- 云端细分类增强

- 进一步细化部分取出与剩余量估计

## 14. 对新对话 / 新成员 / 新 AI 的使用说明

阅读本知识卡时，应默认接受以下事实：

1. 当前项目仍然是比赛阶段性工程原型，不是完整商用智能冰箱系统；
2. 当前主线目标仍然是可靠闭环，而不是盲目追求大模型识别；
3. 事件触发逻辑保留并继续作为第一层；
4. 新增检测、匹配、识别与库存决策分层，是对原架构的增强而非替代；
5. 任何新增方案都不得轻易破坏现有 `event.json` 主接口；
6. 任何扩展都应优先服务于比赛评审得分点；
7. 对外表述时要严格区分“已完成”“预留接口”“后续扩展”，避免叙述失真。
