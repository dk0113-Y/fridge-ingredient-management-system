# 冰箱食材识别与管理系统最终方案基线

## 1. 文档目的

本文档用于固化当前项目的最终开发方向，作为后续编码、联调、测试、答辩材料撰写的统一基线。

本方案严格遵循以下原则：

- 优先完成赛题基础必做项，保证系统闭环可运行。
- 采用“分层识别、逐级确认”思路：先做真实事件检测，再做物体存在性判断，再做粗分类和细分类。
- 面向端侧可落地实现，避免额外训练复杂模型。
- 在当前版本中，仅使用 YOLO 作为目标识别模型；不额外训练专门的部分取出模型。
- 部分取出功能当前仅对果蔬类启用，饮料类、包装食品类、肉蛋生鲜类暂不做部分取出自动识别。

---

## 2. 总体架构

系统划分为 5 个部分：

1. 关键帧与事件窗口提取模块
2. YOLO 粗分类与差异分析模块
3. 大模型细分类与保质期推断模块
4. 出入库与数据库管理模块
5. 主控调度与通信模块

其中第 1、2、4 部分为系统基础主链，第 3 部分为增强识别层，第 5 部分负责整链编排与对外服务。

### 2.1 总体流程

1. 系统在低成本模式下持续监测冰箱门状态与 ROI 变化。
2. 检测到开门并确认发生有效交互后，提取 `before` 与 `after` 关键帧。
3. 对两张关键帧分别运行 YOLO，得到粗分类目标框、类别、数量和置信度。
4. 基于前后帧 YOLO 结果进行数量差分与框级匹配，判定事件类型。
5. 若为放入或取出，则裁剪发生变化的目标区域。
6. 若需要细分类，则将裁剪图送入大模型，获取细粒度名称与置信度。
7. 组合生成标准 `event.json`。
8. 后端解析 `event.json`，更新库存、事件记录和待确认队列。
9. 前端/小程序展示库存、事件、提醒和人工修正结果。

## 3. 模块设计

## 3.1 模块 1：关键帧与事件窗口提取

### 3.1.1 目标

负责从视频流中检测真实冰箱交互事件，并提取用于后续识别的关键帧。

### 3.1.2 输入

- 摄像头视频流或本地测试视频
- ROI 配置
- 门状态检测阈值
- 帧差/亮度变化阈值

### 3.1.3 输出

- `before.jpg`
- `after.jpg`
- `overlay.jpg`
- 事件窗口元信息

### 3.1.4 处理逻辑

1. 常驻运行低成本检测：亮度变化、ROI 帧差、门状态变化。
2. 当冰箱门打开后，进入事件窗口分析阶段。
3. 在交互窗口内寻找一张稳定的 `before` 帧和一张稳定的 `after` 帧。
4. 若窗口变化无效或稳定性不足，则直接输出 `no_change` 或 `uncertain`。

### 3.1.5 说明

本模块只负责确认“是否发生真实事件”和“用于后续分析的关键帧是什么”，不负责最终分类。

---

## 3.2 模块 2：YOLO 粗分类与差异分析

### 3.2.1 目标

对 `before` 和 `after` 两张图分别做粗分类检测，并根据检测结果进行事件判定。

### 3.2.2 粗分类类别

当前统一采用以下 4 个粗分类：

- `fruit_vegetable`：果蔬类
- `meat_egg_fresh`：肉蛋生鲜类
- `drink`：饮料类
- `packaged_food`：包装食品类

### 3.2.3 YOLO 输出

每个检测框输出：

- `coarse_class`
- `confidence`
- `bbox = [x1, y1, x2, y2]`

### 3.2.4 框级匹配

对于前后两帧中同一粗分类的目标，执行匹配：

- 优先按 IoU 匹配
- IoU 不足时按中心点距离匹配
- 匹配成功视为同一位置上的同类目标

匹配后得到：

- `matched_pairs`
- `new_boxes`
- `disappeared_boxes`

### 3.2.5 事件类型定义

当前系统定义以下事件类型：

- `no_change`：无有效变化
- `put_in`：放入
- `take_out`：取出
- `partial_take_out_candidate`：部分取出候选
- `uncertain`：证据不足或复杂场景

### 3.2.6 常规事件判定

1. 若变化占比很低，则判为 `no_change`
2. 若 `new_boxes > 0` 且 `disappeared_boxes == 0`，判为 `put_in`
3. 若 `disappeared_boxes > 0` 且 `new_boxes == 0`，判为 `take_out`
4. 若新增和消失同时较多，或低置信度框比例过高，则判为 `uncertain`

### 3.2.7 低置信度特殊事件

YOLO 识别时设置置信度阈值 `yolo_threshold`。

- 高于阈值：进入正常判定流程
- 低于阈值：进入特殊事件或人工确认流程

低置信度目标信息也必须写入 `event.json`，用于后续调试与复核。

### 3.2.8 部分取出逻辑

当前版本不训练专门的部分取出模型，只在 YOLO 结果基础上用规则判定。

#### 启用范围

`partial_take_out_candidate` 仅对 `fruit_vegetable` 启用。

以下类别当前版本不启用部分取出自动识别：

- `drink`
- `packaged_food`
- `meat_egg_fresh`

#### 判定原则

对于 `fruit_vegetable` 中前后都存在、数量未变化的匹配目标：

- 若框面积变化明显
- 或框内前后裁剪图差异明显
- 或检测框外观形态变化明显

则记为 `partial_take_out_candidate`。

#### 说明

- 大件、可稳定计数的果蔬优先使用数量变化处理。
- 小件、成组、装袋、盒装果蔬才进入部分取出候选判断。
- 饮料类当前版本不做部分取出判断，只做放入、取出、无变化和不确定事件。

### 3.2.9 截图规则

- `put_in`：裁剪 `after` 中新增框
- `take_out`：裁剪 `before` 中消失框
- `partial_take_out_candidate`：同时裁剪 `before` 和 `after` 中匹配框

裁剪时建议对 YOLO 框增加 10%~20% 外扩边距，以保留上下文信息。

## 3.3 模块 3：大模型细分类与保质期推断

### 3.3.1 目标

对模块 2 裁剪得到的目标图进行细粒度识别，并补充保质期信息。

### 3.3.2 定位

本模块是增强识别层，不承担第一道硬判定任务。

### 3.3.3 输入

- 目标裁剪图
- 对应粗分类类别
- 调用模板化 prompt

### 3.3.4 输出

- `fine_name`
- `llm_confidence`
- `expiry_info`（仅包装食品类优先尝试）

### 3.3.5 Prompt 策略

按粗分类分 prompt：

- 果蔬类：输出细粒度名称 + 置信度
- 肉蛋生鲜类：输出细粒度名称 + 置信度
- 饮料类：输出细粒度名称 + 置信度
- 包装食品类：输出细粒度名称 + 置信度 + 是否可根据包装信息推断保质期

### 3.3.6 保质期策略

采用“双轨制”：

1. 对包装食品类，优先尝试从包装信息推断保质期
2. 若无法推断，则回退到本地规则表硬绑定
3. 果蔬类、肉蛋生鲜类、饮料类统一按本地规则表硬绑定

---

## 3.4 模块 4：出入库与数据库管理

### 3.4.1 目标

解析 `event.json`，完成库存更新、事件记录、待确认管理与人工修正。

### 3.4.2 处理流程

1. 读取并校验 `event.json`
2. 根据 `event_type` 生成库存变更建议
3. 若置信度足够高，则自动更新库存
4. 若置信度低、为特殊事件或为复杂场景，则进入待确认队列
5. 用户确认或修正后再正式提交库存表

### 3.4.3 数据库建议表

- `inventory`：当前库存表
- `event_log`：事件历史表
- `pending_review`：待确认事件表
- `shelf_life_rules`：保质期规则表
- `inventory_change_log`：库存变更日志表

### 3.4.4 库存更新原则

- `put_in`：库存增加
- `take_out`：库存减少
- `partial_take_out_candidate`：默认进入人工确认或特定规则处理
- `uncertain`：不直接写入正式库存，进入待确认队列

---

## 3.5 模块 5：主控调度与通信

### 3.5.1 目标

统一编排各模块运行，负责系统联调、接口通信和外部交互。

### 3.5.2 主要职责

- 摄像头/视频流接入
- 事件状态机控制
- 调用关键帧模块
- 调用 YOLO 粗分类模块
- 调用大模型细分类模块
- 生成并分发 `event.json`
- 调用数据库服务
- 向前端/小程序推送库存和事件状态
- 记录日志和异常信息

---

## 4. 最终事件判定策略

### 4.1 主判定顺序

1. 先判断是否发生有效视觉变化
2. 再比较前后帧 YOLO 粗分类结果
3. 再做同类框匹配
4. 优先输出 `put_in` / `take_out`
5. 仅对果蔬类进一步判断 `partial_take_out_candidate`
6. 其余复杂情况输出 `uncertain`

### 4.2 明确约束

- 不额外训练专门的部分取出模型
- 不对饮料类做部分取出自动识别
- 不把“数量不变”直接等同于“无变化”
- 仅在果蔬类中对小件、成组、袋装、盒装场景启用部分取出候选逻辑

## 6. 当前版本的正式结论

### 6.1 已正式确认的方向

1. 项目采用“关键帧提取 + YOLO 粗分类 + 大模型细分类 + 数据库管理 + 主控调度”的总体架构。
2. 当前版本仅使用 YOLO 作为目标识别模型，不额外训练其他专用模型。
3. YOLO 负责粗分类、数量统计、框定位与变化目标提取。
4. 大模型负责细粒度识别与包装食品保质期辅助推断。
5. 系统以 `event.json` 为跨模块协议中心。
6. 后端负责库存闭环、待确认、人工修正和展示。

### 6.2 已正式确认的边界

1. 饮料类当前版本不做部分取出识别。
2. `partial_take_out_candidate` 仅对果蔬类启用。
3. 大件果蔬优先按数量变化处理。
4. 小件、成组、袋装、盒装果蔬才进入部分取出候选判断。
5. 低置信度、强遮挡、多物体混杂等情况进入 `uncertain` 或人工确认流程。

---

## 7. 开发优先级建议

### 第一优先级：基础闭环

- 关键帧提取稳定运行
- YOLO 粗分类跑通
- `event.json` 正常生成
- 后端可解析 JSON 并更新 SQLite
- 前端能展示库存和事件

### 第二优先级：判定增强

- 框级匹配
- 低置信度特殊事件分流
- `uncertain` 与 `pending_review` 流程
- 果蔬类部分取出候选逻辑

### 第三优先级：增强识别

- 大模型细分类
- 包装食品保质期推断
- 小程序联动和视频推流

---

## 8. 答辩表达建议

答辩时可将系统总结为：

“本系统采用事件驱动的分层识别架构。前端首先通过门状态与关键帧分析确认真实交互事件，再通过 YOLO 完成粗分类、数量统计和差异目标提取，随后利用大模型补充细粒度识别与包装食品保质期信息，最终由后端完成库存更新、待确认管理和人工修正，形成完整的冰箱食材识别与管理闭环。为兼顾工程落地性与比赛得分点，当前版本将部分取出功能限定在果蔬类场景，饮料类暂不进行部分取出自动识别。”
